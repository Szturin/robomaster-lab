# Contributing to Robomaster Lab

## 适用范围

这个仓库当前适合团队成员协作维护以下内容：

- 首页、关于我们、小组方向、文档页等静态页面内容
- 外部入口链接（如飞书文档、飞书群、赛事官网等）
- 文档说明与团队维护信息
- 小范围样式优化与移动端可读性调整

当前**不建议**在这个仓库里直接推进：

- 论坛系统
- 用户登录 / 权限系统
- 评论区
- 重型后端功能
- 与当前站点无关的大型前端框架改造

## 协作原则

- 优先最小改动，不为小站点引入无必要复杂度
- 页面内容较小时，优先在现有页面文件内维护，不急于拆组件或抽数据层
- 所有外部链接必须基于已确认地址，不要猜测
- 所有站内链接都要保留 GitHub Pages base：`/robomaster-lab/...`
- 如果只是内容更新，尽量一次只改一个页面或一个主题

## 提交前检查

在提交前，至少完成以下检查：

1. 运行：

```bash
npm run build
```

2. 检查以下事项：
- 站内链接仍使用 `/robomaster-lab/...`
- 外链地址正确且来源明确
- 桌面端与移动端布局没有明显破坏
- 主题切换、导航、页脚等公共区域未被意外影响

## 文档职责分工

- `README.md`：项目入口、运行方式、部署说明、团队维护概览
- `CONTRIBUTING.md`：团队协作与提交建议
- `CLAUDE.md`：AI 助手与前端开发约束

## GitHub Pages 说明

当前仓库已配置 GitHub Pages 自动部署：

- 工作流文件：`.github/workflows/deploy.yml`
- 默认对 `main` 分支的 push 触发部署
- 构建命令：`npm ci` + `npm run build`

因此在合并会进入 `main` 的改动前，建议先本地跑通构建。

## 关于 PR 的建议

如果团队成员共同维护，推荐采用以下轻量流程：

- 开一个小分支处理单一主题改动
- 改完后先本地 `npm run build`
- 提交 PR 时写清楚改了什么、为什么改、怎么验证
- 合并前至少确认页面内容和链接无误

这不是强制的复杂流程，而是为了降低误改首页、导航和公开链接的风险。
