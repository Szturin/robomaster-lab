# robomaster-lab 的飞书内容与多维表格建议

这份文档整理了当前对 `robomaster-lab` 项目接入飞书文档 / 飞书多维表格的建议，目标是：

- 队员不需要修改网站代码
- 官网继续保持 Astro + GitHub Pages 的静态站点模式
- 飞书承担内容维护与协作入口
- 多维表格承担结构化数据源角色

---

## 一、总体建议

建议将飞书内容拆成两层：

### 1. 官网展示层
适合放到网站里做结构化展示的内容：

- 赛事获奖
- 战队介绍
- 成员培养
- 项目展示
- 招新信息
- 对外可公开的技术资料导航

这类内容特点：

- 结构化
- 展示优先
- 更新频率中等
- 适合从多维表格读取

### 2. 内部协作层
继续保留在飞书文档 / Wiki 中的内容：

- 新人培训手册
- 组内开发流程
- 设备借用记录
- 调试记录
- 周报 / 会议纪要
- 详细技术方案
- 内部知识沉淀
- 敏感成员信息

这类内容特点：

- 长文档
- 高频修改
- 协作属性强
- 不适合直接做成官网卡片

一句话：

> 官网负责精选展示，飞书负责内容沉淀；文档负责长文说明，多维表格负责结构化数据。

---

## 二、推荐的飞书组合

### 推荐组合

- **飞书 Wiki**：做内容总入口和长文说明
- **飞书多维表格**：做官网数据源
- **官网**：做精选展示和统一视觉

这个组合最适合战队官网：

- 队员容易维护
- 站点风格统一
- 数据结构清晰
- 后续扩展方便

---

## 三、推荐的飞书信息架构

建议至少建立两套内容：

---

### A. 公开 Wiki

建议名称：

- `RoboMaster Lab 公开资料`
- `科来战队公开文档`

建议目录：

#### 首页
- 战队简介
- 快速入口
- 常用文档导航

#### 成员培养
- 机械组
- 电控组
- 视觉组
- 硬件组
- 运营组

#### 项目与比赛
- 当赛季项目
- 历年比赛成绩
- 复盘资料

#### 开发与协作
- 开发环境
- 仓库说明
- 提交流程
- 常见问题

#### 招新
- 招新时间线
- 方向说明
- 联系方式
- 常见问题

作用：

- 给队员和新成员看
- 给官网提供外链入口
- 做长期内容沉淀

---

### B. 官网数据中心（多维表格）

建议单独建一个多维表格 Base，不要和普通说明文档混在一起。

建议按表拆分：

#### 表 1：赛事获奖
字段建议：

- 年份
- 赛事名称
- 奖项名称
- 参赛项目
- 说明
- 排序
- 是否展示

#### 表 2：项目展示
字段建议：

- 项目名
- 简介
- 负责人/组别
- 状态
- 展示图
- 是否展示
- 排序

#### 表 3：成员培养
字段建议：

- 方向名称
- 培养目标
- 学习内容
- 适合对象
- 是否展示
- 排序

#### 表 4：招新信息
字段建议：

- 标题
- 时间
- 地点
- 报名链接
- 状态
- 是否展示

作用：

- 让官网直接读取结构化数据
- 让不会写代码的成员也能维护内容
- 为后续更多官网模块复用同一套内容系统

---

### C. 内部知识库 Wiki

这一部分和官网数据中心分开。

建议存放：

- 新人入门
- 电控 / 机械 / 视觉培训
- 开发环境安装
- 比赛复盘
- 常见问题
- 调试经验

建议官网只做：

- 摘要
- 导航
- 外链跳转

不建议整块镜像到官网。

---

## 四、官网与飞书的数据关系建议

### 适合放官网的数据

- 获奖成绩
- 战队介绍
- 成员培养
- 招新信息
- 公开项目展示
- 非敏感资料导航

### 不适合直接公开的数据

- 成员手机号
- 内部飞书群 / 内部协作链接
- 比赛策略细节
- 内部仓库地址
- 设备资产信息
- 未公开项目内容
- 太细的调参 / 硬件方案

建议在飞书里就建立“公开层 / 内部层”的区分意识。

---

## 五、字段设计的重要建议

### 1. 字段命名尽量固定

不要今天叫：

- 比赛名称

明天改成：

- 赛事名称

后天又改成：

- 项目名称

如果网站脚本按字段名读数据，字段随意改名会直接导致构建失败或内容缺失。

建议固定一套字段命名，不轻易改。

例如赛事获奖表固定使用：

- 年份
- 赛事名称
- 奖项名称
- 参赛项目
- 说明
- 排序
- 是否展示

---

### 2. 强烈建议加 `是否展示`

用途：

- 草稿数据先录入但不展示
- 未确认信息不直接上官网
- 可控制发布时间

这是最简单也最有效的发布控制字段。

---

### 3. 强烈建议加 `排序`

不要依赖录入顺序控制前端展示。

用途：

- 最新内容置顶
- 重点内容固定前排
- 首页和详情页可共用同一套排序逻辑

建议规则：

- `排序` 数值越小越靠前
- 再结合 `年份` 做二级排序

---

## 六、飞书多维表格接入官网的最小实现建议

推荐采用：

> GitHub Actions 构建时读取飞书多维表格 → 生成本地 JSON → Astro 渲染页面

原因：

- 兼容现有 GitHub Pages 静态部署
- 不暴露飞书密钥
- 实现简单稳定
- 队员只需要维护飞书表格

### 基本流程

```text
飞书多维表格
    ↓
GitHub Actions 调用飞书 API
    ↓
生成 src/data/*.json
    ↓
Astro 页面读取 JSON
    ↓
构建并部署到 GitHub Pages
```

---

## 七、推荐的最小落地步骤

### 第一步
先建一张多维表格：

- `赛事获奖`

先只打通这一张表，不要一开始就接项目展示、成员培养、招新等所有表。

### 第二步
建立飞书开放平台应用，获取：

- `APP_ID`
- `APP_SECRET`
- 多维表格 `APP_TOKEN`
- 表 `TABLE_ID`

### 第三步
GitHub Actions 在构建前执行脚本：

- 读取飞书多维表格记录
- 生成 `src/data/awards.json`

### 第四步
Astro 页面读取 `awards.json`，渲染获奖卡片。

---

## 八、建议的仓库结构

```text
robomaster-lab/
├── scripts/
│   └── fetch-feishu-awards.mjs
├── src/
│   ├── data/
│   │   └── awards.json
│   └── pages/
│       ├── index.astro
│       └── awards.astro   # 可选
└── .github/workflows/
    └── deploy.yml
```

说明：

- `fetch-feishu-awards.mjs`：构建前拉取飞书数据
- `awards.json`：给 Astro 页面直接读取
- `index.astro`：首页展示精选获奖卡片
- `awards.astro`：可选，展示完整获奖列表

## 九、官网中的展示建议

### 首页
建议只展示：

- 3~6 条精选获奖卡片

这样首页不会过长，也更像对外展示站。

### 单独页面
建议做：

- 完整赛事获奖页

这样便于后续扩展历史成绩，不会挤占首页空间。

---

## 十、推荐的第一阶段实施顺序

### Phase 1：数据接通
1. 建飞书多维表格
2. 建飞书开放平台应用
3. 配 GitHub Secrets
4. 写抓取脚本
5. 生成 `src/data/awards.json`

### Phase 2：前端展示
6. 首页或单页渲染获奖卡片
7. 调整卡片样式与移动端布局

### Phase 3：自动化
8. 在 deploy workflow 中接入抓取脚本
9. 测试：改表 → 部署 → 页面更新

---

## 十一、第一版不建议做的事情

为了保持最小实现，第一版不要上来就做：

- 图片上传
- 附件下载
- 前端直接请求飞书开放接口
- 实时更新
- 多表联动
- 复杂权限后台

先把“赛事获奖”这一个闭环跑通。

---

## 十二、结论

对 `robomaster-lab` 来说，最合理的飞书使用方式是：

- **飞书 Wiki**：放说明性、协作性、长文内容
- **飞书多维表格**：放官网结构化数据
- **Astro 官网**：做精选展示和统一视觉

如果从最小落地开始，建议第一步只做：

1. 公开 Wiki
2. `赛事获奖` 多维表格
3. 构建时同步到官网

这样成本最低，也最容易长期维护。
